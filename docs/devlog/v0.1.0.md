# v0.1.0 — from raw mic to AI-cleaned cable

Notes from building Phases 1 and 2. The short version: Prism went from "a
high-pass filter and a gate" to a real AI denoising pipeline you can hear the
difference through, all CPU-only.

## The feedback loop that almost shipped

The very first bug worth telling: if Windows decides the default microphone is
"CABLE Output", Prism would listen to its own output and feed it back into
itself. Forever. The device picker now refuses to ever select a VB-Cable
device as input, no matter what Windows says the default is. Boring fix,
mandatory fix.

## The gate moved, and that mattered

The original plan had the noise gate before the denoiser, straight off the
mic. In practice that was a bad spot for it: the raw mic's noise floor sat
around −35 dBFS, so the gate threshold had to sit above that (−25), and at
that level it chopped the starts of quiet words. Fricatives, soft consonants,
gone.

Moving the gate *after* RNNoise changed everything. On the cleaned signal the
floor is far lower, so the threshold can sit at −45 dBFS and gate true
silence without touching soft speech. The gate also got a hold time so word
endings and brief pauses don't get clipped. Order matters more than tuning.

## RNNoise without the wrapper

The `pyrnnoise` package bundles the RNNoise shared library, but its Python
wrapper drags in an audio-file dependency chain that's broken — and which a
real-time stream doesn't need anyway. So Prism binds straight to the bundled
`rnnoise.dll` with ctypes and never imports the wrapper. The stream runs at
48 kHz with 480-sample blocks, so one block is exactly one RNNoise frame: no
rechunking, no added latency. Measured ~1.3 ms of work per 10 ms block while
speaking, ~0.55 ms on silence.

## DeepFilterNet, the heavier hammer

Phase 2's second denoiser is DeepFilterNet3, and the goal was to add it
without breaking the "least possible footprint" rule. The official package
pulls in PyTorch — hundreds of megabytes for one model. Instead, Prism uses
the torch-free streaming ONNX export running on ONNX Runtime: a ~13 MB CPU
wheel plus a ~13 MB model file.

The export turned out to be a gift: the entire DSP chain (STFT, ERB features,
the network, deep filtering, ISTFT) is baked *inside* the ONNX graph. The
stage feeds one raw 512-sample frame in, gets one enhanced frame out, and
carries twelve recurrent state tensors between calls. A small FIFO rechunks
Prism's 480-sample blocks to 512 and back.

Two honest numbers from measuring it: ~5.7 ms of compute per 10.7 ms frame
(real-time, but about 5× RNNoise), and ~32 ms of algorithmic latency — an
impulse comes out three frames late. That second number forced a subtle fix:
the strength slider blends the dry mic with the denoised signal, and blending
against an undelayed dry path would comb-filter. So the dry path is delayed
by exactly three frames to stay phase-aligned.

## Switching models live

Both denoisers sit behind the same tiny interface (`enabled`, `mix`, one
`process()` call), so the UI can swap them at runtime — the pipeline rebuilds
on the current mic, the same brief restart as switching microphones. If
DeepFilterNet can't load (no onnxruntime, missing model), it falls back to
RNNoise and the UI says so instead of pretending.

## What the meters actually measure

Two readings, both display-only and nearly free. "Room noise" learns the
ambient floor only from frames RNNoise marks as non-speech, so your voice
never inflates it. "Noise removed" is the smoothed gap between input and
output loudness — the look-how-much-junk-it's-killing number. Together they
say: your room is this noisy, and Prism is taking this much of it out.
